# Implement v0.1 IronOps DevOps Claude Plugin Builder

## Task Overview

This task implements v0.1 of the IronOps DevOps Claude Plugin builder against the authoritative specification at `/config/workspace/IronOps/.dev/releases/1.0/0.1/SPEC_IRONOPS_DEVOPS_PLUGIN.md`. The deliverable is a greenfield Python package (`src/ironops/`) located under `/config/workspace/IronOps/` that consumes a YAML manifest declaring upstream sources and per-source file/directory allowlists, shallow-clones each declared upstream Git repository at its resolved default branch HEAD, renders a curated Claude Code plugin tree into a staging directory enforcing co-import and path-safety guards, emits provenance metadata (`META.json`, `THIRD_PARTY_LICENSES.md`, `.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`), validates the rendered staging tree via `claude plugin validate`, and atomically publishes the validated tree to a private GitHub marketplace repository via rsync + git commit/push.

The implementation satisfies all 16 functional requirements (FR-1..FR-16), all 9 non-functional requirements (NFR-1..NFR-9, including the amended NFR-7 with 9 categorical failure codes), and all 10 acceptance criteria (AC-1..AC-10) defined in the spec. Two explicit spec amendments are encoded as discrete checklist items: SPEC §NFR-7 gains the `PUBLISH_FAILED` categorical code (per disposition D3), and SPEC §2.1 plus §17 Definitions are amended to reflect that the builder lives in `src/ironops/` as an installable Python package rather than `scripts/build_plugin.py` (per disposition D4). The task adheres strictly to MDTM Template 02 — every checklist item is a self-contained B2-compliant paragraph, every multi-item process is broken down per A3/A4 granularity, and per-phase QA gates plus a final post-completion validation sequence enforce evidence-based completion.
